## 摘要

新創公司在資源有限、資訊不完整與市場高度不確定的環境中，必須持續回答一個核心問題：有限資金應如何配置，才能形成可持續的獲利能力？研發、行政與行銷支出分別代表產品創新、組織運作與市場擴張，但各項支出對獲利的影響程度並不相同，而且支出與獲利之間可能同時受到產業、公司成熟度、人才品質、募資能力、競爭強度與總體經濟環境影響。若僅依靠直覺判斷，很容易把表面相關誤認為因果關係，或因少數成功案例而高估某一種支出策略。

本白皮書說明一套以 `50_Startups.csv` 為基礎的獲利預測與特徵選擇平台。專案依循 CRISP-DM 六階段方法論，從商業理解、資料理解、資料準備、建模、評估一路延伸至部署與互動式分析。系統不只訓練一個模型，而是建立可重複執行的分析流程，並比較十種特徵選擇方法：Correlation、Mutual Information、Chi-Square、ANOVA F-Test、SelectKBest、Recursive Feature Elimination、Sequential Forward Selection、Lasso、Random Forest Importance 與 XGBoost Importance。每種方法均產生由 Rank 1 到 Rank 5 的特徵排序，再以一致的測試條件比較不同特徵數量下的 RMSE 與 R-squared。

資料集包含五十筆新創公司觀測值，原始輸入欄位為 `R&D Spend`、`Administration`、`Marketing Spend` 與 `State`，預測目標為 `Profit`。資料經數值標準化與州別 One-Hot Encoding 後，形成五個可供排名的模型特徵：三個支出欄位及 `State_Florida`、`State_New York`。探索式分析顯示，`R&D Spend` 與 `Profit` 的 Pearson 相關係數約為 0.9729，遠高於 `Marketing Spend` 的 0.7478 與 `Administration` 的 0.2007。十種特徵選擇方法也一致把 `R&D Spend` 排在第一位，顯示其重要性並非單一演算法的偶然結果。

在固定 `random_state=42`、80% 訓練與 20% 測試的 holdout 實驗中，只使用 `R&D Spend` 的線性迴歸取得最低 Test RMSE 7,714.33 與 Test R-squared 0.9265。十種方法的 Rank 1 都是 `R&D Spend`，因此在單特徵設定下形成並列最佳。另一方面，在全資料的打亂五折交叉驗證比較中，`R&D Spend` 加上 `Marketing Spend` 的平均 CV RMSE 為 8,883.70、CV R-squared 為 0.9389，優於只使用 `R&D Spend` 的 CV RMSE 9,091.21。這兩項結果並不矛盾：固定 holdout 描述某一次切分的表現，而交叉驗證描述不同資料折疊下的平均穩定性。對小型資料集而言，跨折結果通常更適合支援特徵數量決策。

模型比較顯示，Random Forest 在固定測試集取得最佳 Test RMSE 8,629.20 與 Test R-squared 0.9080；Linear Regression 則具有最低的交叉驗證 RMSE 9,550.68，並具備高可解釋性、低複雜度與較佳的小樣本穩定性；XGBoost 在本次資料上表現較弱，Test RMSE 為 12,647.34。這說明更複雜的演算法不必然更準確，尤其當樣本只有五十筆、訊號主要呈現線性關係時，複雜模型可能因資料不足而無法展現優勢。

專案最終提供兩種部署能力。第一種是可載入完整前處理與模型 Pipeline 的 FastAPI 預測服務；第二種是本地端互動式 Dashboard。使用者可調整模型、十種特徵選擇演算法、Top-K 特徵數、測試集比例、Random State、樹模型參數，以及新創公司的支出與州別資料。系統會重新訓練模型並即時顯示 RMSE、R-squared、實際值對預測值散佈圖、十種方法的特徵排名及單筆 Profit 預測。此設計讓專案從靜態報告進一步轉化為可探索、可驗證、可溝通的決策支援工具。

本白皮書的核心結論是：`R&D Spend` 是資料中最一致且最強的獲利預測因子；`Marketing Spend` 在跨折評估中提供有價值的額外資訊；`Administration` 與州別在控制主要支出後的增量預測價值有限。然而，這些結果只能解釋資料中的預測關係，不能直接證明提高研發或行銷預算一定會造成獲利上升。若要把系統提升至真實投資或預算配置場景，仍需擴充資料規模、加入產業與時間維度、建立模型監控、進行公平性與漂移檢查，並將預測結果置於人工審查與商業脈絡之中。

## 一、白皮書目的與適用對象

本白皮書的目的不是只描述一段模型程式，而是完整說明一套從問題定義到互動部署的機器學習決策支援方法。文件同時面向管理決策者、資料分析師、機器學習工程師、軟體開發者、課程評閱者與希望重現實驗的學習者。不同角色可以從本文件取得不同價值：管理者可理解模型能回答與不能回答的問題；分析師可追蹤資料處理與評估邏輯；工程師可理解服務介面與部署結構；學習者可把專案視為一個完整 CRISP-DM 範例。

白皮書採取「證據優先」原則。所有重要結論均應連結至資料特性、模型指標或可重複執行的程式流程，而非僅依靠演算法名稱或視覺印象。當不同評估方法給出不同答案時，本文件不會強行選擇單一敘事，而是說明差異的來源與決策含義。例如，固定 holdout 偏好單一 `R&D Spend` 特徵，而五折交叉驗證偏好 `R&D Spend` 與 `Marketing Spend`；這種差異正是小樣本模型治理必須面對的現象。

本文件所稱的「最佳」必須附帶評估條件。若以固定測試集最低 RMSE 為條件，十種方法選出的第一名特徵均達成相同最佳結果；若以跨折平均 RMSE 為條件，兩特徵組合較佳；若以單次測試集模型表現為條件，Random Forest 較佳；若以跨折穩定性、解釋能力與部署簡潔性為條件，Linear Regression 更適合作為基準部署模型。沒有脫離條件的最佳模型，只有符合特定目標、限制與風險承受度的選擇。

## 二、問題背景：從預算配置到可驗證的決策支援

新創公司的獲利能力通常不是由單一因素決定。產品是否具有差異化、研發速度是否足以形成競爭優勢、行銷是否能有效取得客戶、行政成本是否過度膨脹、團隊是否具備執行能力，以及市場是否處於成長期，皆可能影響最終 Profit。即使如此，支出資料仍具有重要價值，因為支出結構反映公司如何分配稀缺資源，也提供一個可量化、可比較且容易取得的分析入口。

傳統預算討論常落入兩種極端。第一種極端是純粹依賴經驗，認為成功模式可以直接複製；第二種極端是過度依賴模型，把預測數值視為確定結果。前者缺乏一致驗證，後者忽略資料限制與因果不確定性。本專案採取中間立場：把模型視為一個能夠整理證據、比較情境、揭露敏感度與支持討論的工具，而不是取代決策者的自動裁決器。

從商業角度看，本專案希望回答五類問題。第一，哪些支出項目與獲利最有關聯？第二，加入較弱特徵是否真的改善泛化能力？第三，不同特徵選擇方法是否產生一致結論？第四，簡單線性模型與非線性樹模型的效果有何差異？第五，如何把分析結果轉化為容易操作的本地端工具，使使用者能主動改變條件並觀察結果，而不是被動閱讀固定圖表？

這些問題具有普遍性。雖然資料集只有五十筆，欄位也相對簡單，但專案所採用的流程可延伸至行銷預算最佳化、營運成本分析、投資初篩、分店績效預測、專案投報率估計與其他中小型表格資料問題。真正可移植的價值不是某一個 RMSE 數字，而是如何定義問題、保護測試資料、比較方法、解讀差異、部署介面與管理風險。

## 三、商業目標、分析目標與成功標準

### 3.1 商業目標

商業目標是利用可取得的支出與州別資訊，建立新創公司 Profit 的估計工具，協助使用者理解不同資源配置與預測獲利之間的關係。系統應能支援探索，而非只輸出單一答案。使用者需要知道模型選用了哪些特徵、評估表現如何，以及調整資料切分或模型參數後，結論是否仍然成立。

### 3.2 分析目標

分析目標包括：建立完整資料品質檢查；量化各欄位與 Profit 的關係；完成數值與類別資料前處理；比較十種特徵選擇演算法；評估 Top 1 至 Top 5 特徵組合；比較 Linear Regression、Random Forest 與 XGBoost；保存可部署模型；提供 API 與互動式 Dashboard。

### 3.3 成功標準

技術成功標準至少包含以下內容：

1. 訓練與測試流程可重複執行，固定 Random State 時能重現結果。
2. 前處理器僅在訓練資料上擬合，避免測試資料洩漏。
3. 十種特徵選擇方法均能輸出完整排名。
4. 不同 Top-K 特徵組合能以一致模型與指標公平比較。
5. 評估至少包含 RMSE 與 R-squared，並清楚說明其意義。
6. 模型與必要前處理能一起保存並由 API 載入。
7. Dashboard 能接受參數調整、重新訓練並顯示結果。
8. 文件能指出限制、適用邊界與風險，而非只呈現最佳數字。

商業成功標準則更為嚴格。若要在真實企業中使用，系統應證明其預測可改善既有決策流程、降低分析時間、提升預算討論品質，且不會因資料偏差而造成不合理建議。本專案目前屬於教學與原型層級，已具備技術示範價值，但尚未達成生產環境的商業驗證條件。

## 四、資料資產與欄位語意

### 4.1 資料集概況

資料集共有五十筆觀測值與五個欄位，沒有缺失值，也沒有重複資料列。三個支出欄位為連續數值，州別為名目類別，Profit 為連續預測目標。五十筆樣本對機器學習而言非常有限，因此任何結果都可能對切分方式敏感。這項限制影響模型選擇、驗證策略、超參數搜尋與結論強度。

| 欄位 | 類型 | 角色 | 商業解讀 |
|---|---|---|---|
| `R&D Spend` | 數值 | 輸入 | 研究、產品開發、技術與創新投入 |
| `Administration` | 數值 | 輸入 | 管理、人資、法務、財務與辦公營運成本 |
| `Marketing Spend` | 數值 | 輸入 | 廣告、品牌、活動與客戶取得投入 |
| `State` | 類別 | 輸入 | California、Florida 或 New York |
| `Profit` | 數值 | 目標 | 新創公司獲利 |

### 4.2 描述統計

`R&D Spend` 的平均值為 73,721.62，最小值為 0，最大值為 165,349.20；`Administration` 平均為 121,344.64，範圍介於 51,283.14 與 182,645.56；`Marketing Spend` 平均為 211,025.10，最大值達 471,784.10；`Profit` 平均為 112,012.64，範圍從 14,681.40 到 192,261.83。各欄位尺度差異明顯，因此對依賴係數尺度或距離概念的方法而言，標準化是合理處理。

資料中的零支出值被保留。零研發或零行銷不一定是錯誤，可能代表某些公司確實未在該項目投入資金。若在沒有商業證據的情況下把零值視為缺失或異常並刪除，可能反而移除具有重要意義的極端策略。Profit 的潛在離群值同樣被保留，原因是樣本量過小，任意刪除一筆資料都可能顯著改變模型。

### 4.3 州別資料

State 包含 California、Florida 與 New York。州別可能間接反映稅制、人才供應、生活成本、投資人網絡與市場環境，但資料集中沒有這些機制的直接欄位。因此，即使州別特徵出現相關性，也不能解釋為地理位置本身造成獲利差異。州別應視為代理變數，且必須注意代理變數可能攜帶未觀測因素或偏差。

## 五、CRISP-DM 方法論與專案治理

CRISP-DM 將資料探勘專案分為商業理解、資料理解、資料準備、建模、評估與部署六個階段。這六個階段並非只能依序執行一次，而是反覆循環。例如，模型評估可能顯示州別沒有增量價值，促使分析者回到資料理解階段；Dashboard 使用者調整 Random State 後發現結果不穩定，也可能促使團隊重新檢討成功標準。

### 5.1 Step 1：Business Understanding

第一階段將模糊需求轉換為可衡量問題。本專案不是要求模型「找出成功新創」，而是明確定義為使用既有支出與州別資料預測 Profit 的迴歸問題。這個定義同時揭露限制：模型沒有產業、時間、募資、產品、市場規模與團隊資訊，因此不能完整代表新創價值。

### 5.2 Step 2：Data Understanding

第二階段檢查欄位、型態、缺失、重複、分布、離群與相關關係。此階段的核心不是產生漂亮圖表，而是識別後續建模風險。資料只有五十筆，代表模型方差可能很高；資料列可能依 Profit 排序，代表不打亂的交叉驗證可能失真；州別為類別，代表不能直接以任意數字編碼。

### 5.3 Step 3：Data Preparation

第三階段把原始資料轉換成模型可用形式。數值欄位使用 StandardScaler；State 使用 OneHotEncoder，並設定 `drop="first"` 降低完全共線性風險，設定 `handle_unknown="ignore"` 使預測服務遇到未知類別時不會直接崩潰。前處理與模型被封裝在 Pipeline 中，確保相同轉換可在訓練、測試與部署時一致使用。

### 5.4 Step 4：Modeling

第四階段比較線性與非線性模型。Linear Regression 提供簡潔基準與可解釋係數；Random Forest 透過多棵決策樹捕捉非線性與交互作用；XGBoost 以逐步提升方式組合弱學習器。模型複雜度並不是品質的同義詞，必須以資料規模與驗證結果判斷。

### 5.5 Step 5：Evaluation

第五階段不只看單一指標，也檢查評估設計是否符合商業目標。RMSE 以 Profit 同樣單位呈現誤差，且對大型錯誤施加較高懲罰；R-squared 描述模型解釋目標變異的比例。固定測試集適合提供直觀比較，但對五十筆資料可能高度敏感，因此另以打亂五折交叉驗證評估平均穩定性。

### 5.6 Step 6：Deployment

第六階段將模型轉化為可使用服務。專案保存完整 Pipeline，提供 `/predict` 單筆預測 API，並新增 `/api/analyze` 動態分析介面。Dashboard 讓使用者控制模型與特徵選擇設定，將分析從一次性輸出轉為互動式實驗。部署也意味著必須考慮輸入驗證、版本、監控、資安與模型更新，而非只把檔案存成 joblib。

## 六、探索式資料分析與關鍵發現

### 6.1 相關係數

三個數值輸入與 Profit 的相關係數如下：

| 特徵 | 與 Profit 的相關係數 |
|---|---:|
| `R&D Spend` | 0.9729 |
| `Marketing Spend` | 0.7478 |
| `Administration` | 0.2007 |

`R&D Spend` 的相關係數接近 1，表示資料中存在非常強的正向線性關係。`Marketing Spend` 也呈現中高程度正相關，但其訊號較弱。`Administration` 的邊際線性相關有限。需要強調的是，相關係數只描述兩個變數共同變動程度，不代表提高支出一定會造成 Profit 上升，也不排除高獲利公司本來就更有能力投入研發。

### 6.2 分布、尺度與離群

直方圖用於觀察各欄位集中程度與偏態；箱型圖用於檢查四分位距與潛在離群；散佈圖用於觀察支出與 Profit 的線性趨勢；州別圖則比較樣本數與 Profit 分布。圖表顯示 `R&D Spend` 與 Profit 的關係最接近直線，這解釋了 Linear Regression 在小型資料上具有競爭力。

### 6.3 多重共線性與衍生特徵

專案曾考慮 `Total Spend` 與 `R&D Ratio` 等衍生特徵。`Total Spend` 與原始支出欄位高度相關，例如與 Marketing Spend 的相關係數約為 0.9521，與 R&D Spend 約為 0.8697。若同時把總支出與其組成項目放入線性模型，可能引入不必要的多重共線性，使係數解讀不穩定。因此，衍生特徵適合用於額外情境比較，不應在沒有驗證的情況下直接加入最終模型。

### 6.4 小樣本警示

五十筆資料意味著每一筆都具有較高影響力。80/20 切分後，測試集只有十筆，一個特別難預測的公司就可能顯著改變 RMSE。這也是 Dashboard 提供 Test Size 與 Random State 控制的原因：使用者可以親自觀察指標如何隨切分改變，理解單一數字背後的不確定性。

## 七、資料準備與防止資料洩漏

資料洩漏是指模型訓練時直接或間接接觸到預測時不應知道的資訊。若先使用全資料計算標準化平均與標準差，再切分訓練與測試，測試資料的分布資訊便已進入訓練流程；若使用測試集挑選特徵或超參數，再用同一測試集報告最終效果，也會高估泛化能力。

本專案透過 Scikit-Learn Pipeline 與 ColumnTransformer 管理轉換。數值欄位由 StandardScaler 處理，State 由 OneHotEncoder 處理。前處理器在訓練資料上 `fit_transform`，測試資料只執行 `transform`。特徵選擇分析也以訓練資料計算排名，再使用測試資料評估選定特徵組合。這種結構降低了意外洩漏風險，也使部署時能重複使用完全相同的轉換。

One-Hot Encoding 使用第一個州別作為參考類別。經轉換後，模型特徵包含 `State_Florida` 與 `State_New York`，而 California 由兩個欄位都為零表示。這不代表 California 不重要，而是線性模型需要一個基準類別避免虛擬變數完全線性相依。

標準化讓三個支出欄位具有可比較尺度，對 Lasso 係數與某些排名方法尤其重要。樹模型理論上不要求標準化，但在統一分析流程中使用相同轉換，可以讓特徵名稱與資料介面保持一致。需要注意的是，Chi-Square 要求非負輸入且通常用於類別目標；本專案先以 MinMaxScaler 將特徵轉為非負，再將 Profit 分箱後計算 Chi-Square。此結果可作為補充排名，但不應被誤解為標準連續迴歸檢定。

## 八、十種特徵選擇演算法

特徵選擇的目的不是單純刪除欄位，而是評估哪些資訊對預測最有價值、哪些欄位可能增加噪音、不同方法是否形成一致證據。十種方法分屬 Filter、Wrapper 與 Embedded 三大類，各自具有不同假設、成本與解讀方式。

### 8.1 Correlation

Correlation 使用 Pearson 相關係數衡量每個特徵與 Profit 的線性關係。其優點是速度快、容易解釋，適合作為第一層篩選；缺點是只能衡量單變量線性關係，無法捕捉非線性、交互作用或控制其他變數後的增量價值。在本資料中，Correlation 將 `R&D Spend` 排第一、`Marketing Spend` 排第二。

### 8.2 Mutual Information

Mutual Information 衡量兩個變數之間共享的資訊量，可捕捉部分非線性關係。其數值不直接代表係數方向，也可能對樣本量與估計參數敏感。本專案結果顯示 `R&D Spend` 與 `Marketing Spend` 具有最高 Mutual Information，而 Administration 與州別在此切分下接近零。

### 8.3 Chi-Square

Chi-Square 常用於非負特徵與類別目標的獨立性分析。由於本專案目標 Profit 為連續值，因此必須先將 Profit 分箱。這個轉換會損失部分連續資訊，所以 Chi-Square 在此主要作為方法比較與教育用途，而非最適合的迴歸特徵選擇依據。

### 8.4 ANOVA F-Test

ANOVA F-Test／F-Regression 評估單一特徵與連續目標之間的線性關係強度。`R&D Spend` 的 F 分數約為 676.1035，顯著高於 `Marketing Spend` 的 56.6907，其他特徵則接近零。此結果與 Pearson Correlation 高度一致，因為兩者都聚焦單變量線性關係。

### 8.5 SelectKBest

SelectKBest 是一個通用篩選框架，本專案使用 `f_regression` 作為評分函式。它可直接選出最高分的 K 個特徵，容易整合至 Pipeline。由於使用相同 F 分數，排名與 ANOVA F-Test 相同。其價值在於提供標準化選取介面，而非創造新的統計訊號。

### 8.6 Recursive Feature Elimination

RFE 是 Wrapper 方法，反覆擬合模型並移除較弱特徵，直到得到指定數量。它考慮模型在多變量條件下的係數，因此可能與單變量方法產生不同排序。RFE 在本資料中依序偏好 R&D Spend、Marketing Spend、Administration，再到州別特徵。

### 8.7 Sequential Forward Selection

SFS 從空集合開始，每一步加入能使評估指標改善最多的特徵。本專案自行以五折交叉驗證 RMSE 實作前向排名。SFS 的優點是直接對模型效能進行選擇，缺點是計算成本較高，且貪婪策略不保證找到全域最佳組合。在目前訓練資料上，SFS 第二名選擇 Administration，顯示跨折局部決策可能與其他方法不同；然而固定測試集上該兩特徵組合表現較差，提醒使用者不要只依排名名稱判斷品質。

### 8.8 Lasso

Lasso 在線性迴歸損失中加入 L1 正則化，可把弱特徵係數壓縮至零，兼具建模與選擇功能。標準化對 Lasso 很重要，否則不同尺度會影響懲罰。結果中 R&D Spend 的係數絕對值遠高於其他特徵，New York 係數接近零，表示其增量線性訊號有限。

### 8.9 Random Forest Importance

Random Forest 由多棵樹組成，可捕捉非線性與交互作用。特徵重要性反映某特徵在樹分裂中帶來的改善。R&D Spend 的重要性約為 0.9273，Marketing Spend 約為 0.0629，其餘特徵皆很低。傳統 impurity importance 可能偏好連續或可分割點較多的變數，因此解讀時仍需搭配其他方法。

### 8.10 XGBoost Importance

XGBoost 透過梯度提升逐步修正誤差，能處理複雜非線性。其重要性同樣把 R&D Spend 排第一，但第二名為 State_New York，與多數方法不同。這不代表 New York 真正比 Marketing 更重要，而可能反映小樣本中的特定樹分裂。當某一複雜模型的排名與多數方法不一致時，應把差異視為需要調查的訊號，而非直接採納。

## 九、特徵排名結果與一致性分析

十種方法的共同結論非常明確：Rank 1 全部為 `R&D Spend`。這種跨方法一致性比單一分數更具有說服力，因為 Filter、Wrapper 與 Embedded 方法使用不同資訊與假設，卻仍得到相同第一名。對此資料集而言，R&D Spend 是最穩健的預測訊號。

第二名多數為 `Marketing Spend`，但 SFS 選擇 Administration，XGBoost Importance 選擇 State_New York。第三到第五名的差異更大，顯示弱特徵之間的排序不穩定。當特徵本身訊號接近噪音時，演算法細節、資料切分與模型結構都可能改變排序。因此，對排名後段不宜做過度細緻的商業解釋。

| 特徵 | Pearson | Mutual Information | F-Score | RFE Rank | SFS Rank | Lasso 係數 | RF Importance | XGB Importance |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| R&D Spend | 0.9730 | 1.5135 | 676.1035 | 1 | 1 | 37,947.57 | 0.9273 | 0.9361 |
| Marketing Spend | 0.7738 | 1.0505 | 56.6907 | 2 | 3 | 3,457.12 | 0.0629 | 0.0256 |
| Administration | 0.0902 | 0.0000 | 0.3120 | 3 | 2 | -1,736.60 | 0.0069 | 0.0051 |
| State_Florida | 0.0955 | 0.0000 | 0.3497 | 4 | 4 | 493.91 | 0.0017 | 0.0036 |

## 十、Top-K 特徵效能比較

每種演算法都產生五個特徵的完整排序。系統依序取前一到前五個特徵，以相同 Linear Regression 模型在固定測試集上比較。這種設計把「排名品質」與「模型類型」分離，使十種方法能在相同條件下公平比較。

固定 holdout 結果顯示，十種方法的 Top 1 都是 R&D Spend，因此 RMSE 都為 7,714.33，R-squared 都為 0.9265。加入第二個特徵後，多數方法加入 Marketing Spend，RMSE 上升至 8,206.33；XGBoost 加入 State_New York，RMSE 為 7,718.10，僅略差於單特徵；SFS 加入 Administration，RMSE 大幅上升至 9,152.27。當加入全部五個特徵時，各方法最終使用相同資訊，RMSE 收斂至約 9,055.96。

這個現象說明，增加特徵不保證提升泛化能力。弱特徵可能讓模型在訓練資料中擬合額外波動，卻在測試資料造成誤差。對小型資料集，簡潔模型往往具有較低方差。另一方面，單一 holdout 結果也可能只是剛好有利於某個特徵組合，因此不能只根據一次切分永久刪除其他欄位。

專案輸出 `feature_selection_performance_allinone.csv` 保存十種方法、五種特徵數量，共五十組評估結果；兩張綜合圖表呈現 RMSE、R-squared 曲線與 Rank 表格。這些成果不只是報告附件，也可作為後續模型治理與回歸測試基準。

## 十一、交叉驗證與特徵數量決策

為降低單一測試切分的偶然性，專案另外進行打亂五折交叉驗證，依專家排序逐步加入特徵。結果如下：

| 特徵數 | 特徵組合 | CV RMSE | CV R-squared |
|---:|---|---:|---:|
| 1 | R&D Spend | 9,091.21 | 0.9374 |
| 2 | R&D Spend、Marketing Spend | **8,883.70** | **0.9389** |
| 3 | 再加入 State_New York | 9,226.66 | 0.9342 |
| 4 | 再加入 State_Florida | 9,249.63 | 0.9339 |
| 5 | 再加入 State_California | 9,249.63 | 0.9339 |

跨折平均結果偏好兩個特徵，而不是固定 holdout 偏好的單特徵。原因可能是 Marketing Spend 在某些資料折中提供穩定補充訊號，但在 `random_state=42` 的特定測試集沒有改善。對只有五十筆資料的情況，交叉驗證能讓每筆資料輪流參與驗證，通常比一次切分提供更全面的穩定性觀察。

然而，交叉驗證也不是無條件真理。若資料存在時間順序、群組依賴或重複實體，隨機 KFold 可能造成洩漏。此資料集未提供時間與公司群組資訊，因此使用打亂 KFold 是合理教學選擇，但真實應用應根據資料生成方式設計驗證策略。

## 十二、模型選擇與實驗結果

### 12.1 Linear Regression

Linear Regression 假設 Profit 可由特徵線性組合描述。其優點是訓練快速、參數少、容易解釋、對小型資料較穩定。缺點是難以捕捉複雜非線性與交互作用，也可能受離群值與多重共線性影響。本專案中 R&D Spend 與 Profit 的關係高度線性，使 Linear Regression 成為強而合理的基準。

### 12.2 Random Forest

Random Forest 透過多棵樹的平均降低單棵樹方差，可捕捉非線性與特徵交互作用。固定測試集上，它取得三個模型中最低 Test RMSE 8,629.20 與最高 Test R-squared 0.9080。然而其 CV RMSE 9,667.17 略差於 Linear Regression，且模型解釋較不直接。

### 12.3 XGBoost

XGBoost 常在大型表格資料競賽表現優異，但其能力依賴足夠資料與適當調參。本專案中，XGBoost 的 CV RMSE 11,625.48、Test RMSE 12,647.34，顯著弱於前兩者。這並不表示 XGBoost 是差演算法，而是顯示演算法與資料條件必須匹配。當樣本少、主要關係近似線性時，提升樹容易擬合局部噪音。

### 12.4 模型比較表

| 模型 | CV R-squared | CV RMSE | Test R-squared | Test RMSE |
|---|---:|---:|---:|---:|
| Linear Regression | 0.9340 | **9,550.68** | 0.8987 | 9,055.96 |
| Random Forest | **0.9360** | 9,667.17 | **0.9080** | **8,629.20** |
| XGBoost | 0.9005 | 11,625.48 | 0.8025 | 12,647.34 |

### 12.5 部署模型選擇

專案保存 Multiple Linear Regression Pipeline 作為基準部署模型，原因包括：跨折 RMSE 最低；結果容易向非技術利害關係人解釋；小樣本下較不容易因超參數而劇烈變動；前處理與模型可一併保存；預測速度快。Random Forest 可作為候選 challenger 模型持續監控；XGBoost 則適合作為資料量增加後重新評估的候選。

## 十三、評估指標的意義與誤用風險

RMSE 是平方誤差平均後開根號，單位與 Profit 相同。它對大誤差較敏感，適合在大型預測錯誤具有較高成本時使用。例如，若投資決策無法容忍把低獲利公司高估為高獲利公司，RMSE 能對此類錯誤給予較大懲罰。但 RMSE 容易受離群值影響，也無法直接告訴使用者誤差方向。

R-squared 描述模型相對於以平均值預測能解釋多少變異。R-squared 0.9265 表示該測試集上約 92.65% 的 Profit 變異被模型解釋，但不代表每一筆預測都接近真實值，也不代表模型具有因果解釋能力。R-squared 會受測試資料分布影響，在小測試集上可能不穩定。

模型比較時不應只挑一個最漂亮的指標。若 Random Forest 在單次測試集勝出，但 Linear Regression 在跨折 RMSE 與解釋能力上更有優勢，決策者應根據用途選擇。高風險決策可能偏好穩定、透明且易審查的模型；純預測服務可能偏好經充分驗證後的最低誤差模型。

## 十四、互動式 Dashboard 設計

本地端 Dashboard 的目的，是讓模型評估從靜態報告變成可操作實驗。使用者可選擇 Linear Regression、Random Forest 或 XGBoost；選擇十種特徵排序方法之一；設定 Top 1 至 Top 5 特徵；調整測試集比例與 Random State；修改樹模型的 Estimators、Max Depth 與 Learning Rate；輸入新創公司的 R&D、Administration、Marketing 與 State。

每次按下重新分析按鈕，FastAPI 後端會重新讀取資料、切分訓練與測試、在訓練資料上執行十種特徵選擇、取得指定方法的 Top-K 特徵、訓練指定模型、計算測試指標，並對使用者輸入資料預測 Profit。前端顯示四個核心指標、實際值對預測值散佈圖、目前選定特徵、預測 Profit，以及十種方法的完整排名表。

Dashboard 特別適合展示模型敏感度。使用者可改變 Random State，觀察最佳特徵與 RMSE 是否變動；切換 Test Size，理解測試樣本增加如何影響訓練資料量；將模型切換至 XGBoost，觀察複雜模型不一定改善結果；增加 Top-K，理解更多特徵可能使測試表現下降。這種互動能把「模型不確定性」轉化為可見經驗。

目前 Dashboard 使用原生 HTML、CSS、JavaScript 與 SVG，不依賴外部 CDN 或前端框架。優點是部署簡單、離線可用、依賴少；限制是大型資料、複雜圖表與狀態管理能力有限。若未來擴充，可考慮 React、Vue、Plotly、Streamlit 或專業 BI 工具，但應先確認新增複雜度是否帶來實際價值。

## 十六、決策解讀與商業建議

### 16.1 R&D Spend

R&D Spend 是所有方法共同的第一名，且單獨使用即可在固定測試集取得最佳結果。這表示研發投入是本資料集中最有價值的預測訊號。管理者可把研發效率、產品開發成果與創新投資回報視為優先分析主題。但不能因此直接建議無限制增加研發預算，因為資料只顯示相關，且沒有衡量研發品質、時滯與邊際報酬。

### 16.2 Marketing Spend

Marketing Spend 在多數排名中位居第二，且在五折交叉驗證中與 R&D Spend 組合取得最佳平均 RMSE。這表示行銷可能提供跨樣本較穩定的額外訊號。管理建議應聚焦行銷投資報酬、客戶取得成本、轉換率與留存率，而不是只追求支出規模。

### 16.3 Administration

Administration 與 Profit 的邊際關係很弱，多數方法也將其排在後段。行政支出通常支援組織運作，而非直接創造收入。管理者應關注行政效率與必要性，但不宜簡化為「行政支出越低越好」，因為良好治理、人才制度與法遵能力也可能支援長期成長，只是目前資料未能呈現。

### 16.4 State

州別特徵的增量價值有限。這可能表示控制支出後，三州差異不足以改善預測；也可能因樣本太少，無法估計地理效果。州別不應作為主要投資篩選條件，更不能被用作缺乏證據的地域偏好。若要研究地理因素，應加入稅率、人才成本、租金、市場規模與創投密度等更直接變數。

### 16.5 使用方式

模型適合支援預算情境比較、分析討論與教育展示，不適合單獨決定投資、貸款、資源裁撤或人員決策。合理流程是：模型提出訊號，分析師檢查資料與假設，業務專家補充脈絡，決策者綜合風險後判斷，最後追蹤結果並更新模型。

## 十七、風險、限制與倫理治理

### 17.1 樣本量限制

五十筆資料不足以代表所有新創公司。模型可能只學到特定資料來源、時期或產業結構。任何高精度數字都應搭配信賴區間、重抽樣或不同切分敏感度分析。若未來取得更多資料，應重新驗證所有排名與模型結論。

### 17.2 遺漏變數

Profit 可能受到產業、成立年數、員工數、融資階段、產品成熟度、競爭、利率、景氣、客戶集中度與管理品質影響。模型未包含這些欄位，因此現有支出係數可能吸收遺漏因素。預測可用不代表解釋完整。

### 17.3 因果誤用

預測模型回答「在資料中，哪些條件與 Profit 同時出現」，不回答「若主動增加某支出，Profit 會增加多少」。要做因果推論，需要實驗、自然實驗、工具變數、傾向分數或其他因果設計。把相關模型直接轉為預算處方可能造成錯誤決策。

### 17.4 過度擬合與模型選擇偏差

若反覆查看測試集並根據結果調整模型，測試集會逐漸成為訓練決策的一部分。專案目前已比較多種方法，因此最佳 holdout 數字可能帶有選擇偏差。正式研究應保留最終未觸碰測試集，或使用 Nested Cross-Validation。

### 17.5 公平性與代理變數

State 可能是社經條件的代理變數。即使其預測價值有限，也應評估模型是否對不同區域產生系統性誤差。若未來加入更多公司或個人層級資料，更應避免把受保護屬性或其代理用於不公平決策。

### 17.6 資安與隱私

目前資料不包含個人敏感資訊，但真實企業財務資料通常具有高度機密性。正式部署需加入身份驗證、傳輸加密、存取控制、稽核紀錄、資料保存政策與秘密管理。Dashboard 的動態訓練端點也應限制使用，以防止資源濫用。

### 17.7 模型漂移

市場環境、成本結構與商業模式會變化。即使模型目前有效，未來也可能因資料分布改變而失準。生產系統需監控輸入分布、預測分布、實際誤差、特徵重要性與不同群組表現，並設定再訓練與人工審查門檻。

## 十八、可重現性、測試與品質保證

可重現性來自固定資料、明確相依套件、Random State、完整 Pipeline、輸出產物與版本控制。`requirements.txt` 記錄 pandas、scikit-learn、matplotlib、Pillow、joblib、xgboost、FastAPI、uvicorn 與 httpx 等依賴。訓練腳本可重新產生 EDA、特徵排名、效能 CSV、圖表、模型與中繼資料。

品質保證至少應包含四層。第一層是語法與靜態檢查，確保程式可解析；第二層是資料契約檢查，確認欄位、型態與允許值；第三層是模型回歸測試，確認固定設定下指標沒有非預期大幅改變；第四層是 API 與介面測試，確認首頁、健康檢查、動態分析與預測服務可用。

本專案已透過實際 HTTP 服務驗證 Dashboard 首頁與 `/api/analyze`。例如切換至 XGBoost 模型、XGBoost 排名、Top 3、30% 測試集與自訂參數後，服務能回傳不同特徵、RMSE、R-squared 與 Profit 預測。這證明介面不是靜態展示，而是真正驅動後端重新分析。

未來應加入自動化測試，包括：不同模型與演算法組合的參數化測試；非法輸入回應測試；未知州別測試；極端值與零支出測試；模型檔不存在時的降級行為；Dashboard JavaScript 的端對端測試；API 效能與併發測試。

## 二十、未來發展路線

### 20.1 資料擴充

最高優先事項是增加資料量與欄位深度。建議加入產業類別、成立年份、員工數、募資階段、營收、毛利率、現金流、客戶數、留存率、研發人員比例、廣告轉換率、地區成本與時間序列。資料擴充可讓模型回答更具商業價值的問題，也有助於區分支出規模與支出效率。

### 20.2 驗證策略升級

可導入 Repeated K-Fold、Nested Cross-Validation、Bootstrap 信賴區間與時間切分。Nested CV 可降低在同一資料上選特徵與評估造成的樂觀偏差；Bootstrap 可估計指標與排名的不確定範圍；若資料加入時間維度，應改用時間序列驗證避免未來資訊洩漏。

### 20.3 模型與解釋

可增加 Ridge、Elastic Net、Huber Regression、Gradient Boosting、CatBoost 與 Bayesian Regression。解釋層可加入 Permutation Importance、Partial Dependence、SHAP 與單筆預測解釋。模型解釋應聚焦「模型如何使用資訊」，而非宣稱因果。

### 20.4 MLOps

正式系統可加入 MLflow 或其他實驗追蹤、模型登錄、資料版本、CI/CD、自動測試、容器化、監控與警報。每次模型更新都應保存資料版本、程式版本、參數、指標、核准者與回滾方式。Dashboard 可顯示目前 production 與 challenger 模型比較。

### 20.5 決策支援升級

預測只是第一步。未來可建立情境模擬與最佳化模組，例如在總預算限制下比較不同 R&D 與 Marketing 配置。但最佳化必須承認模型不是因果模型，否則演算法可能利用相關性產生不可靠建議。較安全做法是把模擬結果標示為探索性情境，並設定合理範圍與人工審查。

## 二十一、治理原則與建議政策

第一，所有預測必須附帶模型版本、資料版本、主要輸入與評估條件。第二，不得將 Profit 預測作為唯一投資或資源分配依據。第三，任何涉及州別或其他代理變數的決策都應進行公平性檢查。第四，模型使用者必須理解預測與因果的差異。第五，模型表現應定期重新評估，若 RMSE 超過門檻或資料漂移顯著，應暫停自動使用並啟動審查。

第六，動態 Dashboard 的分析結果應被視為實驗，不應自動覆蓋 production 模型。第七，超出訓練資料範圍的輸入應顯示警示。第八，應保存重要決策的輸入、預測、人工判斷與最終結果，以便稽核與改善。第九，對外溝通不得只宣稱高 R-squared，而應同時揭露樣本量、誤差與限制。第十，當資料不足以支持可靠結論時，系統應允許回答「不確定」。

## 二十二、最終結論

本專案展示了如何把一個小型迴歸資料集發展成完整的機器學習分析與互動部署流程。最重要的成果並不是某一個模型分數，而是建立了可重現的 CRISP-DM 工作流、跨十種方法的特徵排名比較、固定 holdout 與交叉驗證的差異解讀、模型複雜度的實證比較，以及可供使用者主動探索的本地端 Dashboard。

證據一致指出 R&D Spend 是最強的 Profit 預測因子。Marketing Spend 在多數方法中排名第二，並在五折交叉驗證中提升兩特徵組合的平均表現。Administration 與 State 的增量價值有限，且排名容易受方法影響。固定 holdout 最佳結果為只使用 R&D Spend 的 RMSE 7,714.33 與 R-squared 0.9265；跨折特徵數量比較則偏好 R&D Spend 與 Marketing Spend。Random Forest 在單次測試集表現最佳，Linear Regression 在跨折 RMSE、透明度與部署穩定性之間取得較佳平衡。

任何模型結論都必須放回資料限制中理解。五十筆資料不足以支援高風險自動化決策，相關性不能取代因果推論，單次測試結果也不能代表長期穩定性。正確使用方式是把模型當成結構化證據工具：幫助使用者提出更好的問題、比較不同假設、辨識敏感度、記錄實驗並支援人工判斷。

若未來能取得更大量、更多維度與具時間性的資料，並建立嚴謹驗證、模型監控與治理機制，本平台可以從教學原型逐步演進為更可靠的商業決策支援系統。無論技術如何升級，透明、可驗證、可重現與尊重不確定性，都應維持為系統核心原則。

## 附錄 C：術語表

**CRISP-DM：** 跨產業資料探勘標準流程，包含商業理解、資料理解、資料準備、建模、評估與部署。  
**Regression：** 預測連續數值的監督式學習問題。  
**Feature：** 模型用於預測的輸入欄位。  
**Target：** 模型欲預測的目標欄位，本專案為 Profit。  
**Pipeline：** 把前處理與模型串接成一致流程的物件。  
**One-Hot Encoding：** 將名目類別轉換為多個 0/1 欄位。  
**StandardScaler：** 將數值轉換為以平均與標準差為基礎的標準尺度。  
**Data Leakage：** 訓練流程不當使用預測時不可取得的資訊。  
**Holdout Test：** 保留一部分資料作為固定測試集。  
**Cross-Validation：** 透過多次資料折疊評估模型平均表現。  
**RMSE：** 均方根誤差，越低越好，單位與目標相同。  
**R-squared：** 決定係數，描述模型解釋目標變異的程度。  
**Filter Method：** 不依賴最終模型或只用統計量進行特徵選擇。  
**Wrapper Method：** 反覆訓練模型以比較特徵子集。  
**Embedded Method：** 在模型訓練過程中同時形成特徵重要性或選擇。  
**Random State：** 控制隨機操作的種子，使結果可重現。  
**Hyperparameter：** 訓練前設定的模型參數，例如樹深與估計器數量。  
**Model Drift：** 模型因環境或資料分布改變而逐漸失準。  
**Challenger Model：** 與目前 production 模型比較的候選模型。  
**Causal Inference：** 估計介入行為是否造成結果改變的方法。

## 附錄 D：審查清單

### 資料審查

- 資料來源是否合法、可信且可追溯？
- 欄位定義是否一致？
- 缺失、重複、零值與離群是否已檢查？
- 訓練資料是否代表實際使用情境？
- 是否包含敏感屬性或代理變數？

### 模型審查

- 前處理是否只在訓練資料上擬合？
- 特徵選擇是否避免使用測試資訊？
- 是否比較簡單與複雜模型？
- 是否同時報告 holdout 與跨折結果？
- 是否檢查不同 Random State 的敏感度？
- 是否保留最終未觸碰測試集？

### 部署審查

- 輸入是否有型態、範圍與允許值驗證？
- 模型與前處理是否版本化？
- 是否記錄預測與模型版本？
- 是否有監控、警報與回滾方式？
- 是否限制動態訓練端點的存取？
- 是否明確說明模型不可作為唯一決策依據？

### 溝通審查

- 是否避免把相關性描述為因果？
- 是否揭露樣本量與資料限制？
- 是否說明「最佳」所使用的評估條件？
- 是否同時呈現誤差與解釋能力？
- 是否讓非技術使用者理解不確定性？
